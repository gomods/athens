本文档阐述了希望彼此合作的方式。社区中可能出现各种情况，很难为每种情况都制定规则——这基本等同于预测未来！

我们当然会设定一些界限，如[行为准则](https://github.com/gomods/athens/blob/main/CODE_OF_CONDUCT.md)，但我们希望在遇到需要解决的新情况或问题时能回到本文档寻求指导。

# 指导原则

这是整个文档的摘要！Athens 项目有几个指导原则：

- 彼此友善
- 简化开发和测试
- 关注社区
- 提出问题

在本文档的其余部分，我们将详细讨论每个项目。

# 彼此友善

我们坚信，社区应以友好、热情和建设性为优先，代码和技术次之。如果社区成员彼此不友善，技术再酷也没用。

- 我们让新手感到受欢迎
- 我们在 GitHub 问题中进行辩论，并保持文明和建设性
- 我们感同身受
- 我们[多听少说](https://hbr.org/2017/02/in-a-difficult-conversation-listen-more-than-you-talk)
- 我们在不同时区，所以尊重这一点
- 我们互相鼓励学习项目的新方面

最重要的是，让我们**包容**。不是每个人都会认同您的观点、沟通风格和许多其他方面。尝试考虑他们的观点并尊重他们。

# 简化开发和测试

编写代码时认知负担是有害的，所以让我们尽量减少它。

- 入门应该是一行命令搞定
- 困难的事情要让它变得简单
- 文档很重要，要保持更新
- 写出让人乐于使用的代码

# 关注社区

有一句非洲谚语是这样说的：

>如果你想走得快，独自走。如果你想走得远，一起走。

我们希望将这种智慧应用于我们的社区。

- 自己完成一切既简单又诱人
- 但如果我们想让项目持续增长，只有几个人做所有事情是很困难的
- 相关：[总线因素](https://en.wikipedia.org/wiki/Bus_factor)
- 所以让我们专注于将新人引入社区，并让他们开始参与（见上文：彼此友善）

# 提出问题

提问是分享想法、改进项目和社区的好方式。

- 如果您不知道某事，尽量不要害怕提问
- 如果您认为您的问题很愚蠢，仍然要问
- ... 如果您在公开场合提问仍然不舒服，请私下向维护者提问
- 如果有人问您问题，稍后回答也没关系
- 如果可以的话，把答案写下来让每个人都能看到
- ... FAQ 很值得拥有——让我们做吧！
- 新手有最好的视角，所以认真倾听他们的问题
